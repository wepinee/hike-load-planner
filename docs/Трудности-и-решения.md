# Трудности реализации и способы их преодоления

Ниже приведён фрагмент для пояснительной записки (раздел «Проблемы и решения» / «Особенности реализации»). Объём — около одной страницы.

---

## 1. Модуль распределения нагрузки (наиболее сложный блок)

**Суть задачи.** Предметная область требует учитывать три различных типа груза: общее снаряжение (делится между участниками), личное снаряжение (закреплено за владельцем) и питание (рассчитывается как доля на каждого участника с учётом длительности похода и числа людей). Кроме расчёта необходимо заранее проверить допустимость данных: наличие участников, заданные лимиты веса, указанные владельцы личных предметов, непревышение суммарной ёмкости группы.

**Трудности.** Первая — согласование модели `Assignment` с реальными назначениями: для записей, относящихся только к еде, поле `gear_item_id` должно оставаться пустым, иначе СУБД отклоняет вставку (`Column 'gear_item_id' cannot be null`). Вторая — разделение алгоритмов по типам без «разрастания» одного сервиса: потребовались отдельные стратегии распределения, фабрика создания назначений и набор правил-спецификаций (Policy), запускаемых до расчёта. Третья — представление результата для пользователя: сводка по участникам, список назначений, ручное переназначение только общего снаряжения.

**Решение.** Схема таблицы `assignments` приведена в соответствие с логикой (nullable-колонки, патч при старте приложения). Расчёт вынесен в `DistributionService` с цепочкой `DistributionValidator` и стратегиями в пакете `service.distribution`. На клиенте (React) вкладка «Раскладка» отображает итог и графики, не дублируя бизнес-правила.

---

## 2. Целостность данных при удалении похода

**Трудность.** При удалении похода вместе с участниками, снаряжением, едой и планами загрузки возникала ошибка Hibernate о «устаревшей» сущности `Participant`: после массового удаления связанных записей контекст persistence содержал несогласованное состояние.

**Решение.** В `HikeService.delete()` удаление выполняется в явном порядке (назначения → планы → снаряжение → еда → участники), с вызовами `flush()` и `clear()` у `EntityManager` перед финальным `deleteById` похода. Это гарантирует актуальность кэша первого уровня и успешное завершение транзакции.

## 3. Клиентское приложение и аутентификация JWT

**Трудность.** Изначально интерфейс строился на Thymeleaf с сессионной авторизацией. Для современного SPA потребовался отдельный фронтенд (React, TypeScript, Vite) и stateless API: выдача JWT при входе и регистрации, передача токена в заголовке `Authorization`, настройка CORS и прокси разработки `/api` → Spring Boot. Дополнительная работа — перенос всех вкладок карточки похода (участники, снаряжение, питание, раскладка, «моя раскладка») на единый REST-клиент.

**Решение.** Реализованы `JwtService`, фильтр в цепочке Spring Security, `AuthContext` на React, хранение токена в `localStorage`, защищённые маршруты `ProtectedRoute`. Пользовательский сценарий демонстрируется на порту 5173; сервер остаётся источником данных и правил.

## 4. Разграничение прав организатора и участника

**Трудность.** Не все операции должны быть доступны любому авторизованному пользователю: меню похода и общее снаряжение ведёт организатор, участник может добавлять только личное снаряжение (при привязке email к записи в списке участников). Несогласованность UI и API приводила бы к ошибкам «Только организатор может выполнить это действие».

**Решение.** На сервере — метод `ensureOrganizer`, отдельные эндпоинты `gear/shared` и `gear/personal`, проверка владельца при изменении личного предмета. На клиенте — флаг `isOrganizer`, условный рендеринг форм; для участника отображается блок «Добавить личное снаряжение» и редактирование только своих строк в таблице.

---

## 5. Вспомогательные задачи (меньшая сложность, заметные затраты времени)

- **Валидация форм:** согласование `@Valid` на DTO (дата старта не в прошлом, длина имени и пароля) с проверками на фронтенде и понятными сообщениями вместо общей фразы «Ошибка валидации».
- **Интерфейс:** выравнивание таблиц при разном числе колонок (просмотр vs редактирование), центрирование диаграмм Recharts, русские подписи перечислений (роль, статус, тип снаряжения).
- **Расчёт питания:** итоговый вес и калории на весь поход считаются на сервере; важно было не смешать в отчёте «на человека» и «на группу».

---

## Вывод

Наибольшая сложность проекта связана не с отдельными CRUD-экранами, а с **сквозной бизнес-логикой раскладки** и её согласованием с базой данных, API и клиентом. Второй по значимости блок — **миграция на React с JWT** и **ролевая модель** доступа. Преодоление перечисленных трудностей обеспечило работоспособность демонстрационного сценария: от создания похода до формирования нагрузки, визуального контроля лимитов и экспорта результата.
